# Регламент соревнований (черновик прототипа)

**Редакция документации: 1.7.7.**

## Когда читать

- **Обязательно:** по необходимости.
- **Когда:** если нужен полный регламент, tie-break и юридически точные правила сдачи.
- **Можно пропустить:** да, если вы идёте по `contest_task.md` и `quickstart_2days.md`.

## Термины

- **SGA** (*security goals and assumptions*) — в коде и API; в текстах допустимо **ЦПБ** (цели и предположения безопасности).
- **Регулятор** — сертификация пакета АБУ; **ЦР** — парк установок и миссии.

## Последовательность действий конкурсантов

1. Прочитать **[contest_task.md](contest_task.md)** (задание, user story) и **[quickstart_2days.md](quickstart_2days.md)**.
2. Развернуть окружение: `make install`. Рекомендуется **Linux** или **GitHub Codespaces** / **WSL2** (см. [README.md](../README.md#платформа-linux-и-окружение)).
3. Изучить [context.md](context.md), [architecture.md](architecture.md), [certification_process.md](certification_process.md).
4. Разрабатывать решение в `src_solution/` (или согласованный каталог), опираясь на `src_starting_point/`. Структура АБУ решения: подкаталоги `src_solution/abu/tcb` (ДВБ) и `src_solution/abu/other` (прочий код); CycloneDX решения — `src_solution/sbom/SBOM_TCB.cdx.json` и `SBOM_OTHER.cdx.json`.
5. Прогонять `make tests-all` и процедуру сертификации (`make prepare-cert-bundle`, `make certify-abu` или API Регулятора).
6. Заполнить обязательный **отчёт о решении** в каталоге `src_solution/docs/` (ориентируйтесь по шаблону [`src_starting_point/docs/solution.md`](src_starting_point/docs/solution.md); в корне `docs/solution.md` — только краткий указатель) и при сдаче приложить **таблицу баллов** по шаблону [templates/evaluation_report.md](templates/evaluation_report.md).
7. Подать работу: **ссылка на приватный онлайн-git-репозиторий** с решением; в репозиторий добавить **пользователя, указанного организаторами**, с правами **разработчика** (GitHub: *Collaborator* с доступом на запись к репозиторию; GitLab: роль **Developer** или выше — по инструкции жюри), чтобы жюри могло клонировать и проверить код.

## Запрещённые приёмы

- Подмена или отключение автоматических тестов заготовки без обоснования в отчёте.
- Обход Регулятора (подделка сертификата, недоступный для проверки хэш).
- Публикация секретов, ключей, персональных данных в репозитории.
- Плагиат без ссылки на источник.

## Критерии оценки

Оцениваются **прежде всего изменения конкурсанта** (`src_solution/`, тесты, сопровождаемые артефакты). Критерии, связанные **только** с оценкой свойств учебных подсистем **ЦР** и **Регулятора** как отдельных объектов заслуги участника, **не используются**; вместо них введены сквозной автотест сценария **ЦР–АБУ** и критерии по решению в `src_solution/`.

- **25 критериев** (**C01…C25**). Каждый оценивается **от 0 до 3 баллов** (целые и дробные уровни внутри критерия заданы в [criteria_rubric.md](criteria_rubric.md)); **номинальная сумма raw — 75** баллов (см. `make evaluate-score` и [scripts/evaluate_contest_score.py](../scripts/evaluate_contest_score.py)).
- Автоматическая часть: тесты (в т.ч. сквозной сценарий ЦР–АБУ), артефакты сдачи, эвристики по структуре `src_solution/` (**C18–C19**, **C23–C25**).
- Экспертная часть: **C20** — место в рейтинге по **стоимости сертификации** (3 / 2 / 1 / 0 балла за 1–3-е места, остальные 0); **C21–C22** — жюри по архитектуре/политикам и по полноте отчёта (в автоматическом скрипте — **0**, итог вносится в отчёт вручную).

**Имя функции в файле теста pytest не является строкой таблицы баллов:** итог по Cxx выводит `make evaluate-score` и шаблон отчёта.

### Перечень критериев (имена как в скрипте оценки)

| № | Кратко |
|---|--------|
| C01 | Все тесты репозитория (включая тесты решения) завершаются успешно |
| C02 | Все тесты решения находятся в подкаталогах `src_solution/tests/**` |
| C03 | Маркер `security` в `pytest.ini` и использование в тестах |
| C04 | Тесты журнала / `event_log` |
| C05 | Пример `sga.json` |
| C06 | Примеры CycloneDX `SBOM_TCB` / `SBOM_OTHER` в `src_starting_point/sbom/` |
| C07 | Успешное выполнение сертификации решения (сборка пакета + успешный ответ Регулятора) |
| C08 | Сквозной автотест **ЦР–АБУ** (`tests/test_e2e_abu_dm_scenario.py`) |
| C09 | Оформление кода в `src_solution/` (flake8, PEP8): 0 замечаний — 3; 1–5 — 2; иначе 0 |
| C10 | Решение: журнал `event_log` в `src_solution/` |
| C11 | Решение: зависимости в `src_solution/` (`requirements.txt` / `pyproject.toml`): тяжёлые в ДВБ → 0, вынос в `requirements-other.txt` при пустом `requirements.txt` → 3 |
| C12 | Тесты репозитория **импортируют** `src_solution` (AST) |
| C13 | В отчёте в каталоге `src_solution/docs/` есть раздел по тестам безопасности и привязка к коду в `src_solution/` |
| C14 | **`numpy`** в SBOM решения (`src_solution/sbom/`): только в **SBOM_OTHER** — до 3; в **SBOM_TCB** — 0 (см. [criteria_rubric.md](criteria_rubric.md#c14)) |
| C15 | В тестах под `tests/` есть **импорты** и журнала **`event_log`**, и **`src_solution`** |
| C16 | Покрытие **`src_solution/abu/tcb`** тестами: ≥80% — 3; 60–79% — 2; 40–59% — 1; <40% — 0 |
| C17 | Отчёт в каталоге `src_solution/docs/` (ориентир: `src_starting_point/docs/`; см. указатель [`docs/solution.md`](solution.md)) |
| C18 | Решение: **security_monitor**, **policies** в коде; тесты политик (в т.ч. под `src_solution/`) |
| C19 | Разнесение доменов **процессами** (выполнение); каталоги `abu/tcb`/`other` дополняют анализ покрытия; при отсутствии процессной изоляции — единая граница доверия к коду (см. [criteria_rubric.md](criteria_rubric.md#c19)) |
| C20 | **Стоимость сертификации:** 3 / 2 / 1 / 0 за 1–3-е место по возрастанию стоимости (жюри после сравнения всех участников; в скрипте — 0) |
| C21 | **Экспертно:** соответствие политик безопасности **новой архитектуре** АБУ (жюри; в скрипте — 0) |
| C22 | **Экспертно:** полнота отчёта и воспроизводимость (жюри; в скрипте — 0) |
| C23 | Размер домена ДВБ (`src_solution/abu/tcb`): оценка по максимальному LOC одного домена |
| C24 | Количество интерфейсов домена ДВБ: число разрешающих IPC-политик на домен |
| C25 | Наличие `security_monitor` в решении, security-тесты и покрытие monitor-кода (порог 60%) |

## Разрешение равенства баллов

1. Больший балл за **автоматическую** часть (в т.ч. сумма по C01–C19 и C23–C25, где применимо).  
2. При равенстве — учёт баллов по **C20–C22** (стоимость сертификации, экспертные критерии).  
3. Более **ранняя** успешная сдача в окне приёма.  
4. Меньше **предупреждений/ошибок** в отчёте сертификации.  
5. **Жюри** по фиксированному чек-листу (порядок объявляется до старта).
