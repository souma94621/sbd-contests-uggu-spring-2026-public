# Частые вопросы участника

## Когда читать

- **Обязательно:** по необходимости.
- **Когда:** после `quickstart_2days.md`, если есть вопросы по критериям и проверке.
- **Можно пропустить:** да, если всё выполняется по quickstart.

## Где должны лежать тесты решения?

Тесты решения размещаются только в `src_solution/tests/**`.  
Если тесты решения найдены вне этого каталога, снижается `C02`.

## Как подтвердить process-изоляцию?

Нужны признаки разнесения доменов по процессам в коде и тестах (`multiprocessing`, `DomainProcess` или эквивалент), а не только раскладка файлов по каталогам. Это проверяется в `C19`.

## Как не потерять баллы по C07?

`C07` оценивает не наличие скрипта, а факт успешной сертификации:

```bash
make prepare-cert-bundle-solution
make certify-abu-solution
```

Нужен успешный ответ Регулятора и корректный сертификат.

## Как не потерять баллы по C23–C25?

- `C23`: не раздувать доверенные домены; держать их небольшими и простыми.
- `C24`: минимизировать количество разрешающих IPC-политик для каждого доверенного домена.
- `C25`: иметь `security_monitor`, security-тесты и покрытие monitor-кода не ниже порога.

## Что считается накруткой?

- Формальное разделение каталогов без process-изоляции.
- Широкие политики «разрешить всё», маскирующие отсутствие контроля.
- Отчёт без привязки к реальному коду и тестам.
- Перемещение тестов решения вне `src_solution/tests/**` ради обхода критериев.

Такие практики снижают оценку по смысловым критериям (`C18`, `C19`, `C24`, `C25`, `C13`).

## Сокращения

- **АБУ** — автономная буровая установка.
- **ЦР** — цифровой рудник.
- **ДВБ** — доверенная вычислительная база.
- **SG** — цели безопасности.
- **SA** — предположения безопасности.
- **SGA** — совокупность целей и предположений безопасности.
- **IPC** — межпроцессное взаимодействие.
- **SBOM** — ведомость компонентов программного обеспечения.
- **TCB** — доверенная часть кода в составе SBOM.
- **OTHER** — недоверенная часть кода в составе SBOM.
- **СКИБ** — система с конструктивной информационной безопасностью.

## Методическая основа СКИБ

Для конкурсного задания СКИБ означает следующее:

- безопасность закладывается в архитектуру, а не добавляется после реализации;
- критичные функции выделяются в малую доверенную базу, которая дробится на сравнительно небольшие простые домены безопасности, с минимально возможным количество интерфейсов;
- взаимодействия между доменами разрешаются только по явным политикам безопасности, которые применяются монитором безопасности;
- всё, что критично для целей безопасности, должно быть проверяемо тестами и сертификацией;
- стоимость и сложность сертификации снижаются за счёт минимизации доверенной базы в целом и отдельных доменов безопасности в частности, а также количества интерфейсов для доверенных доменов безопасности.

### Примеры из `references`: как использовать

- `references/isolation/README.md`
  - **Зачем:** освоить базовый шаблон process-изоляции.
  - **Что прочитать:** краткое описание архитектуры, обязательные негативные проверки, чек-лист переноса.
  - **Как работать:** запустить тесты примера, затем повторить структуру в своём коде.
  - **Когда можно не использовать:** если у вас уже доказана изоляция доменов по процессам.

- `references/secure_ipc/README.md`
  - **Зачем:** получить рабочий образец политик IPC и контроля маршрутов.
  - **Что прочитать:** формат политик, таблицу соответствия шагов, анти-паттерны.
  - **Как работать:** взять идею минимального белого списка и теста блокировки недопустимого запроса.
  - **Когда можно не использовать:** если у вас уже есть монитор безопасности, политики и покрытие тестами.

- `references/traffic_light_demo/cyberimmunity-traffic-lights-example.ipynb`
  - **Зачем:** понять общую идею доменной архитектуры и посредничества монитора безопасности на компактном примере.
  - **Что прочитать:** вводную часть, диаграммы и шаги взаимодействия компонентов.
  - **Как работать:** перенести принципы разделения ответственности и явного контроля маршрутов в собственную архитектуру.
  - **Когда можно не использовать:** если архитектурные решения и политики уже обоснованы TARA и подтверждены тестами.
