# Метод TARA для конкурсной задачи

## Когда читать

- **Обязательно:** по необходимости.
- **Когда:** если нужно обосновать модель угроз в отчёте и связь архитектуры с целями безопасности.
- **Можно пропустить:** да, если вы идёте только по базовому маршруту из `quickstart_2days.md`.

## Роль TARA в решении задачи

TARA (анализ угроз и оценка рисков) нужен, чтобы:

- определить, какие компоненты критичны для целей безопасности;
- обосновать границу доверенной вычислительной базы;
- показать, какие угрозы закрываются архитектурой, политиками и тестами;
- связать технические решения с критериями оценки и сертификацией.

TARA в конкурсе не требует большой теории. Нужна короткая, проверяемая и воспроизводимая модель с явными выводами.

## Методика выполнения анализа

### Шаг 1. Выделить активы и границы

- Определите активы: люди, оборудование, команды управления, телеметрия, журнал событий, настройки.
- Отметьте границы доверия: внешний контур, внутренние домены, интерфейсы обмена.

### Шаг 2. Зафиксировать цели безопасности

- Выпишите 2–4 цели безопасности (ЦБ/SG) и предположения (ПБ/SA).
- Уточните, какие последствия считаются недопустимыми.

### Шаг 3. Построить 1–2 пути атаки

- Для каждого пути укажите:
  - источник угрозы;
  - уязвимую точку;
  - шаги развития атаки;
  - потенциальный ущерб.

### Шаг 4. Оценить риск

- Дайте качественную оценку вероятности и ущерба (низкий/средний/высокий).
- Укажите, какие угрозы требуют обязательного закрытия в архитектуре.

### Шаг 5. Привязать меры защиты

- Для каждого значимого пути атаки укажите контрмеры:
  - изоляция доменов;
  - политики взаимодействия и принцип «запрещено по умолчанию»;
  - тесты безопасности;
  - артефакты сертификации и SBOM.

### Шаг 6. Зафиксировать результаты в отчёте

- Краткая таблица «угроза -> риск -> мера -> проверка тестом».
- 1 диаграмма или текстовая цепочка атаки (по необходимости).

## Примеры (универсальные)

### Пример 1. Подмена внешнего запроса управления

- **Актив:** целостность критичной команды.
- **Путь атаки:** внешний запрос с некорректными параметрами -> попытка пройти в доверенный домен.
- **Ущерб:** выполнение опасного действия.
- **Меры:** фильтрация на границе, проверка политик IPC, тест блокировки недопустимого запроса.

### Пример 2. Компрометация зависимости недоверенного контура

- **Актив:** корректность входных данных для доверенного решения.
- **Путь атаки:** компрометированная библиотека искажает данные -> доверенный модуль получает ложный вход.
- **Ущерб:** неверное решение в критичном сценарии.
- **Меры:** минимизация доверенной базы, разделение SBOM, проверка граничных условий тестами безопасности.
