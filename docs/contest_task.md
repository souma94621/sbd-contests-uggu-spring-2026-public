# Конкурсное задание (кратко)

Документ **1.7.7**. Читайте его **первым**, затем [quickstart_2days.md](quickstart_2days.md), после чего откройте [criteria_rubric.md](criteria_rubric.md) (баллы C01–C25).

## A. Пользовательский сценарий

**Как** разработчик АБУ, участвующий в отборе поставщика ПО для цифрового рудника,  
**я хочу** привести учебный прототип АБУ к требованиям Регулятора (ЦПБ, ДВБ, SBOM, тесты, сертификация),  
**чтобы** система проходила автоматические проверки, получала осмысленный сертификат и была описана в отчёте.

**Ограничения:** рабочий код решения — в [`src_solution/`](../src_solution/) (структура `abu/tcb`, `abu/other`, `sbom/`); опирайтесь на [`src_starting_point/`](../src_starting_point/). Не подменяйте тесты заготовки без обоснования в отчёте.

**Базовая линия заготовки.** В репозитории уже есть полностью работающая связка: сборка сертификационного пакета из `src_starting_point`, прохождение проверок Регулятора и сквозной сценарий «ЦР → АБУ» (см. корневые тесты `tests/`). Без разделения каталогов `abu/tcb` и `abu/other`, без изоляции доменов процессами и без переноса тяжёлых зависимостей (например numpy) из доверенной базы в недоверенную зону стоимость сертификации по модели Регулятора завышена — в том числе из‑за множителя для «тяжёлых» зависимостей в SBOM ДВБ. Задача участника — улучшить архитектуру, снизить TCB и цену сертификации, не ломая тесты (критерии C14, C16, C18–C19, C23–C25, [certification_process.md](certification_process.md)).

## C. Что нужно сделать (формулировка задания)

1. **Разделить доверенный и недоверенный код** в решении: каталоги `src_solution/abu/tcb` (ДВБ) и `src_solution/abu/other`, зафиксировать зависимости и **SBOM** (`src_solution/sbom/SBOM_TCB.cdx.json`, `SBOM_OTHER.cdx.json`).
2. **Обеспечить журналирование и политики** в духе регламента: журнал событий, при необходимости `security_monitor`, `policies`, изоляцию доменов **процессами** (куда идеально ложится `multiprocessing`/`DomainProcess`; каталоги `tcb/other` упрощают покрытие; см. C18–C19).
3. **Покрыть ДВБ тестами** и поддерживать прохождение всех тестов репозитория, включая тесты безопасности и сквозной сценарий ЦР–АБУ.
4. **Пройти процедуру сертификации** учебного пакета (скрипты `make prepare-cert-bundle`, `make certify-abu` или API Регулятора).
5. **Оформить отчёт** в каталоге `src_solution/docs/` (шаблон — [`src_starting_point/docs/solution.md`](../src_starting_point/docs/solution.md), указатель в корне — [`docs/solution.md`](solution.md)) и таблицу баллов по [templates/evaluation_report.md](templates/evaluation_report.md).

**Содержательная работа** — проектирование и обоснование архитектуры, политик, трассировка целей безопасности к тестам, воспроизводимые команды. Простой **перенос файлов** из `src_starting_point` в `src_solution` без осмысленных изменений и тестов не заменяет эти пункты (см. уровни 1–3 в регламенте по каждому Cxx).

## E. Баллы и тесты pytest

Итоговые баллы по критериям выставляет скрипт `make evaluate-score` ([`scripts/evaluate_contest_score.py`](../scripts/evaluate_contest_score.py)) и при необходимости жюри (C20–C22). **Имена функций в файлах тестов и вспомогательные комментарии в коде не являются строкой таблицы баллов** — ориентируйтесь на вывод `evaluate-score` и шаблон отчёта.

Единый набор команд самопроверки:

```bash
make install
make tests-all
make evaluate-score
make certify-abu-solution
```

## F. Платформа

**Эталонная среда для проверки жюри — Linux** (команды из [Makefile](../Makefile), пути POSIX). Разработка на **Windows** возможна (в т.ч. **WSL2**, **Docker**, **GitHub Codespaces**); без Linux-совместимой среды возможны проблемы с `make`, shell и путями. Рекомендуется прогнать `make tests-all` в среде, близкой к Linux, перед сдачей. Подробнее — в [README.md](../README.md#платформа-linux-и-окружение).
